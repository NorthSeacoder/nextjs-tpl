# Spec: Bootstrap Next.js Template

## Background

当前仓库要定义一个可复用的 Next.js 全栈模板，用于后续快速启动单用户、自部署、agent-friendly 的 Web 产品。

现阶段已有一份设计草案，已明确两个关键方向：

- 数据库默认复用同服务器公共 PostgreSQL，而不是每个项目单独起库容器
- 优先复用 NAS 公共基础设施，避免每个项目重复建设通用运行组件
- 需要参考成熟 Next.js templates 的优点，但避免把模板做成重型 SaaS 脚手架

本 spec 的目标是把这次模板建设的需求边界固定下来，作为后续 `clarify / plan / tasks / implement` 的输入。

## Goal

交付一个可运行、可复制、可自部署的 Next.js 模板基座，使新项目可以在较少改动下直接开始业务开发。

该模板必须优先解决以下问题：

- 提供统一的 Web + API + Database 基础结构
- 提供单用户登录和受保护页面基础能力
- 提供面向公共 PostgreSQL 的数据库接入方式
- 提供优先复用 NAS 公共基础设施的部署路径
- 提供最小但完整的 CRUD 示例链路
- 提供适合 NAS / 单机 / 自部署环境的运行与部署路径

## Target Users

- 个人开发者
- 维护低用户量内部工具的开发者
- 需要让 agent 通过 API 访问系统的项目维护者
- 需要在已有服务器、NAS 或反向代理环境中快速落地应用的人

## User Scenarios

### Scenario 1: 新项目初始化

开发者复制模板后，配置环境变量、连接公共 PostgreSQL、执行初始化命令，即可启动一个带登录、dashboard、示例 CRUD 和 API 的基础应用。

### Scenario 2: 单用户自部署

开发者将模板部署到已有服务器，只启动 Web 服务，连接服务器上已有公共 PostgreSQL，即可完成运行，不需要为每个项目额外维护独立数据库容器。

### Scenario 3: 作为后续业务项目基座

开发者保留模板的认证、数据库、API 和测试结构，在此基础上替换示例业务页面和数据模型，快速进入真实业务开发。

### Scenario 4: agent 访问

后续项目或自动化 agent 可以基于模板提供的受保护 API 和契约文档接入，而不必从零定义认证与接口组织方式。

## Functional Requirements

### 1. Repository Foundation

- 模板必须提供清晰的项目结构，覆盖 Web、共享配置、数据库层和 API 契约层。
- 模板必须作为“基础模板”而不是“完整 SaaS 产品”来组织内容。
- 模板必须包含最小可运行页面和 API，而不是只提供空目录。

### 2. Routing and Pages

- 模板必须提供公开首页。
- 模板必须提供登录页。
- 模板必须提供至少一个受保护的登录后首页。
- 模板必须提供设置页或等价系统页，用于承载账号/API token/系统信息类能力。
- 模板必须提供示例 CRUD 页面，用于验证表单、列表、数据库和 API 链路。

### 3. Authentication

- 模板必须支持单用户优先的登录流程。
- 模板必须支持受保护页面访问控制。
- 模板必须支持登出。
- 模板必须支持首次启动后创建或注入管理员账号。
- 模板第一版必须包含 PAT 管理能力，并提供完整管理入口。

### 4. Database Usage

- 模板必须默认使用同服务器公共 PostgreSQL。
- 模板不得把“每个项目单独启动 PostgreSQL 容器”作为默认路径。
- 模板必须支持通过环境变量配置数据库连接。
- 模板必须支持项目级隔离，避免不同项目共享同一组表对象。
- 模板必须以“共享 database + 项目独立 schema”作为默认隔离策略。
- 模板应允许文档说明独立 database 作为可选替代策略，但不是默认实现路径。
- 模板必须支持 migration 和 seed 仅作用于当前项目范围。
- 模板可以提供本地独立数据库 fallback，但该模式只能是备用方案，不能成为主路径。

### 5. API Foundation

- 模板必须提供健康检查接口。
- 模板必须提供登录态相关接口。
- 模板必须提供最小示例资源的 CRUD API。
- 模板必须为后续 agent / 外部客户端调用预留公共 API 组织方式。
- 模板必须包含可维护的接口契约产物。
- 模板第一版必须提供 API token 的创建、查看、撤销所需接口与页面入口。

### 8. Repository Shape

- 模板第一版必须采用 monorepo/workspace 结构。
- 模板结构必须支持将数据库层、Web 应用、共享配置和 API 契约分离组织。
- 模板不得以“先做单应用仓库，后续再拆”为默认落地方式。

### 6. Developer Experience

- 模板必须提供明确的环境变量样例。
- 模板必须提供本地开发命令。
- 模板必须提供数据库相关命令。
- 模板必须提供基础 lint、typecheck、test 入口。
- 模板必须提供 README 或等价文档说明最短启动路径。
- 模板应说明自动构建镜像的触发分支、目标 registry 和所需环境变量。

### 7. Deployment

- 模板必须支持 Docker 化运行。
- 模板部署路径必须兼容已有 Traefik 或等价反向代理环境。
- 模板部署策略必须优先复用 NAS 上已存在的公共基础设施。
- 模板生产部署默认只要求部署 Web 服务并接入已有公共 PostgreSQL。
- 模板必须保留本地可直接执行的 Docker 构建与运行路径。
- 模板应提供基于 GitHub Actions 的自动镜像构建与推送能力，用于稳定分支（如 `release`）推送时触发。
- GitHub Actions 中的镜像构建与推送属于自动化增强能力，不替代本地和服务器侧的直接构建能力。

## Non-Functional Requirements

- 模板必须偏向 self-host first。
- 模板必须遵循 shared infra first，优先复用 NAS 公共基础设施而不是为每个项目重复启动通用服务。
- 模板必须保持轻量，避免在第一版引入大量可选基础设施。
- 模板必须优先考虑后续复用性，而不是满足某个具体业务。
- 模板必须让新项目的上手路径足够短。
- 模板文档必须清楚区分“模板基础能力”和“后续可扩展能力”。

## Out of Scope

- 多租户体系
- 企业级权限系统
- OAuth / SSO
- 复杂 RBAC
- Redis、队列、worker、对象存储等默认基础设施
- 支付、订阅、计费
- 原生移动端
- 大规模实时协作
- 完整业务域模型

## External Reference Requirements

后续设计与实现阶段应吸收成熟 Next.js templates 的以下经验：

- 保持极短的启动路径
- 提供最小闭环，而不是只给空骨架
- 区分核心模板能力与扩展能力
- 将认证、受保护路由、基础数据链路和测试样例作为模板一等能力

不要求复制任何外部模板的完整功能集合。

## Acceptance Criteria

- 存在一套明确的模板规格，可指导后续设计与实现。
- 后续实现完成后，开发者应能基于模板启动一个带登录、受保护页面、示例 CRUD、基础 API 的应用。
- 模板的默认数据库路径应明确指向公共 PostgreSQL，而非项目私有数据库容器。
- 模板的文档应明确说明公共数据库模式下的隔离方式和使用边界。
- 模板的默认隔离方式应明确为共享 database 下的独立 schema。
- 模板的部署说明应明确 shared infra first 原则及其适用边界。
- 模板第一版应包含可用的 API token 管理页及对应基础接口。
- 模板应支持在约定稳定分支推送时自动构建并推送镜像。
- 模板的范围应保持在“可复用工程基座”而不是“重型 SaaS starter”。
