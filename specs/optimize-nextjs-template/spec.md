# Spec: Optimize Next.js Template

## Background

当前模板已经具备基础的 App Router、登录、受保护页面、示例 CRUD 和 API 能力，但从 Next.js 最佳实践角度看，默认工程基线还有几处明显偏差：

- 一些本应由 Server Components 直接处理的数据读取，被做成了整页客户端拉取
- 认证态与用户信息存在重复读取
- App Router 常规边界文件不完整
- 站内导航仍混用原生 `<a>`，没有统一采用 Next.js 导航约定
- 根布局在 metadata 和字体优化方面仍偏弱
- 登录表单仍采用较传统的客户端提交模式，未体现 Next.js 当前推荐的同源表单处理方式
- 部分 Route Handler 的输入边界缺少明确校验

这些问题虽然不一定导致模板立即失效，但会让模板的默认架构偏离 Next.js 16 / App Router 的推荐方向。由于该仓库定位是“可复用模板”，这些默认选择会被后续项目继承，因此需要把模板基线调整到更符合最佳实践的状态。

本 spec 的目标，是把这轮“按 Next.js 最佳实践优化模板”的需求边界固定下来，供后续 `clarify / plan / tasks / implement` 使用。

## Goal

将当前模板优化为一个更符合 Next.js App Router 最佳实践的工程基座，使其在以下方面具备更稳健的默认基线：

- 服务端优先的数据获取与渲染组织方式
- 更清晰的 Server / Client 边界
- 更完整的 App Router 路由约定与错误/加载兜底
- 更标准的站内导航与根布局基础设施
- 更符合 Next.js 当前推荐方向的同源表单与认证交互方式
- 更明确的 API 输入边界

优化后的模板应仍保持轻量、可复用和易于理解，不应因为追求“最佳实践”而演变为重型框架。

## Target Users

- 复制该模板作为新项目起点的开发者
- 希望模板默认结构更贴近 Next.js 16 推荐写法的维护者
- 需要在后续业务项目中直接复用认证、页面结构和 API 组织方式的开发者

## User Scenarios

### Scenario 1: 新项目开发者直接复用模板页面结构

开发者复制模板后，默认拿到的页面结构应体现“服务端优先、客户端最小化”的 App Router 组织方式，而不是整页客户端化的示例。

### Scenario 2: 开发者在模板基础上扩展受保护页面

开发者新增 dashboard 页面时，应能沿用统一的认证态获取方式、边界文件组织方式和内部导航方式，而不是在多个页面中重复读取用户信息或自行补齐错误/loading 约定。

### Scenario 3: 开发者复用登录流程

开发者复用模板中的登录页时，应看到一个符合当前 Next.js 推荐方向的同源表单处理模式，便于后续扩展，而不是继续复制传统的“客户端状态 + 手写 fetch”模式。

### Scenario 4: 开发者扩展内部 API

开发者以现有 Route Handler 为参考新增接口时，应能继承清晰的输入边界和错误响应约定，减少“直接透传 request.json()”这类松散模式。

## Functional Requirements

### 1. Server-First Page Baseline

- 模板中的示例受保护页面必须体现 Server Components 优先的数据读取方式。
- 首屏即可在服务端拿到的数据，不应默认依赖整页客户端组件在 hydration 后再发起内部 API 请求。
- 模板应明确区分“首屏数据读取”与“局部交互逻辑”的边界。
- 模板中的示例页面必须为后续项目提供可复用的 server-first 参考，而不是把整页 `'use client'` 作为默认模式。

### 2. Clear Server / Client Boundaries

- 模板必须减少不必要的整页客户端化。
- 仅在确有交互需求的局部区域使用客户端组件。
- 同一请求链路中的用户会话或基础用户信息，不应在 layout 和 page 中无意义重复读取。
- 模板应让后续开发者容易识别哪些内容属于服务端职责，哪些内容属于客户端职责。

### 3. App Router Route Conventions

- 模板必须补齐必要的 App Router 边界文件约定。
- 对用户可感知的错误态、404 和 loading 态，模板必须提供明确的默认落点或组织方式。
- 模板中的路由结构应能作为后续页面扩展的参考，不要求开发者每次手动补齐基础边界。
- 若某类边界文件在当前模板中不适用，spec 后续阶段必须能说明不适用的原因，而不是缺失后保持隐式状态。

### 4. Internal Navigation Conventions

- 模板中的站内导航必须默认采用 Next.js 推荐的内部导航方式。
- 公开页、dashboard 布局和其他模板内示例页面，不应继续把普通站内跳转实现为原生文档导航。
- 外部链接、文件下载或明确需要 document navigation 的场景可继续使用原生链接，但不应作为模板的站内导航默认值。

### 5. Root Layout Baseline

- 根布局必须提供比当前更完整的 metadata 基线。
- 模板必须提供适合作为项目起点的标题、描述和站点级 metadata 组织方式。
- 模板必须采用 Next.js 推荐的字体优化方式，避免把字体加载完全留给后续项目自行补齐。
- 根布局优化后，仍应保持模板中立，不应引入强业务语义或过度装饰性的品牌设定。

### 6. Same-Origin Form and Auth Flow Baseline

- 模板中的登录流程必须更贴近 Next.js 当前推荐的同源表单处理模式。
- 优化后，登录页应能作为后续业务项目的参考实现，而不是停留在传统 SPA 风格的表单提交模式。
- 登录流程仍必须保留清晰的错误反馈和成功后的导航结果。
- 此次优化只聚焦模板内已有单用户登录流程，不扩展到多用户认证体系、OAuth 或外部身份提供商。

### 7. Route Handler Input Boundary

- 模板中的 Route Handler 必须体现明确的输入边界。
- 对 JSON body、路径参数和必要字段的处理，模板必须提供更清晰的约束和错误响应方式。
- 示例接口应能作为后续扩展 API 的参考基线，减少宽松输入直接下传到 service 层的情况。
- 输入边界优化不要求在本 spec 中规定具体校验库或实现形式，但要求后续实现具备一致、明确的校验行为。

## Non-Functional Requirements

- 优化后模板必须继续保持轻量，不引入与目标无关的复杂抽象。
- 模板必须优先体现 Next.js App Router 推荐实践，而不是兼容旧版 Pages Router 心智。
- 模板的默认示例必须易读、易改，适合作为新项目复制基础。
- 优化不得破坏现有模板的核心能力：登录、受保护页面、示例 CRUD、API token 管理、基础 API。
- 模板不得因为优化而显著增加理解门槛。

## Out of Scope

- 引入新的业务功能
- 重做视觉设计或大规模改版 UI
- 引入多租户、RBAC、OAuth、SSO
- 引入 React Query、SWR 或其他新的客户端数据层作为默认依赖
- 设计完整的前端状态管理方案
- 替换数据库、ORM 或认证模型
- 对所有 API 做大规模协议升级
- 引入复杂的 observability、analytics 或性能监控系统

## Acceptance Criteria

- 模板中的示例受保护页面不再把整页客户端拉取作为默认数据读取模式。
- 模板中的用户会话/基础用户信息读取方式得到收敛，不再存在明显重复读取的默认实现。
- `app` 目录具备明确的错误、404、loading 等基础路由边界组织方式，足以作为模板默认参考。
- 模板中的站内导航默认遵循 Next.js 推荐方式，而不是继续使用普通原生链接作为主路径。
- 根布局具备更完整的 metadata 基线，并包含 Next.js 推荐的字体优化方式。
- 登录流程能够体现更现代的同源表单处理模式，同时保留错误反馈与成功跳转能力。
- 示例 Route Handler 具备更明确的输入边界和错误响应行为。
- 优化完成后，模板仍保持轻量，不因“最佳实践”而演化为重型 starter。
- spec 中不存在会阻塞后续规划的关键歧义。
